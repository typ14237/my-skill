# 通用规范技能

这是一个基于 [forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills) 核心思想整理的通用工程规范技能，用于约束代理在日常软件开发中保持清晰、克制和可验证。

## 适用场景

- 新功能开发
- 缺陷修复
- 局部重构
- 代码审查
- 测试补充
- 验证与收尾

## 核心原则

- 先澄清，再实现
- 先简单，再扩展
- 只改必要内容
- 用验证证明完成

## 1. 先澄清，再编码

在开始实现前，不要默认自己已经完全理解需求。

执行要求：
- 先明确任务目标、输入、输出和边界。
- 把关键假设写出来，不要默默猜测。
- 如果需求存在多种合理解释，先指出分歧点，再选择方案。
- 如果当前方案明显过重、过慢或风险过高，要主动提出更简单的替代方案。
- 如果信息不足以支撑可靠决策，应先补齐上下文，再进入实现。

自检问题：
- 我能否用一句话说清这次改动要解决什么问题？
- 我能否明确指出这次任务不包含什么？

## 2. 简单优先

只写解决当前问题所需的最少代码，不为想象中的未来需求预支复杂度。

执行要求：
- 不添加用户未要求的能力。
- 不为一次性逻辑抽象出额外层级。
- 不为了“以后可能会用”而提前加入配置、扩展点或模式设计。
- 不为了极小概率场景堆砌防御性分支。
- 如果实现显得绕，优先思考能否删减后仍满足目标。

常见反模式：
- 用过度通用的结构包装简单逻辑。
- 把简单需求拆成过多文件、类型或接口。
- 在没有证据时提前优化性能、复用性或灵活性。

自检问题：
- 是否存在更短、更直接但同样可靠的实现？

## 3. 手术式改动

只修改完成任务所必需的部分，不顺手重写无关内容。

执行要求：
- 不因为看见旧代码不顺眼就顺手重构。
- 不为了个人偏好修改命名、格式或注释。
- 不做与任务目标无关的“顺便优化”。
- 尽量延续原有结构和项目约定，除非它直接阻碍当前任务。

如果本次改动带来连带影响：
- 清理由本次改动造成的无用导入、变量、分支或函数。
- 对于本来就存在但与本任务无关的问题，只标记风险，不主动扩大改动范围。

自检问题：
- 我的每一行变更，是否都能直接解释为完成这次任务所必须？

## 4. 目标驱动执行

把任务改写成可以验证的结果，而不是停留在模糊动作。

执行要求：
- 在实现前或实现时同步定义成功标准。
- 能写测试时，优先通过测试表达目标。
- 不能写自动化测试时，也要给出明确的手工验证路径。
- 不把“代码写完”当作“任务完成”，而是以验证通过作为完成标准。

示例转换：
- “修复问题” -> “先复现问题，再让复现用例通过”
- “增加校验” -> “先补失败场景验证，再实现校验逻辑”
- “重构模块” -> “保持行为不变，并完成关键路径回归验证”

推荐计划格式：

```text
1. [步骤]
   verify: [如何确认本步骤完成]
2. [步骤]
   verify: [如何确认本步骤完成]
3. [步骤]
   verify: [如何确认整体目标达成]
```

自检问题：
- 最终有什么证据可以证明任务已经完成？

## 5. 风险显式升级

遇到不确定性、冲突或隐藏成本时，不要隐式绕过，要明确暴露。

需要升级沟通的情况：
- 需求之间存在冲突。
- 改动会影响公共接口、兼容边界或已有行为。
- 当前实现依赖强假设。
- 发现用户目标与现有架构明显不匹配。
- 缺乏足够验证手段，无法建立可靠信心。

沟通要求：
- 先说事实，再说风险，再给建议。
- 给出一到两个现实可行的方案。
- 明确标出自己依赖的关键假设。

## 6. 收尾检查清单

在结束任务前，至少快速检查：
- 这次实现是否比必要的更复杂？
- 是否修改了与任务无关的内容？
- 是否验证了关键路径和潜在回归？
- 是否留下了由本次改动产生的无用代码？
- 是否可以用一句话总结验证结果？

## 建议输出结构

使用本技能时，最终输出尽量包含：
- 任务理解或关键假设
- 简短执行计划
- 实际完成的改动
- 验证方式与结果
- 剩余风险或待确认事项

## 一句话口诀

**先澄清，后实现；先简单，后扩展；只改必要内容；用验证证明完成。**
